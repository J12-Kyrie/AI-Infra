### 原理

![](C:\Users\86184\AppData\Roaming\marktext\images\2025-07-31-17-08-05-image.png)

1. 根据上述设计思路，我们实现了第一个版本的 CUDA 归约核函数 `reduce_v0`。从代码中可以看出，我们通过一个循环逐步完成归约操作。以第 9 行开始的 `for` 循环为例：在第一次迭代中，当线程索引 `tid == 0` 时，它会将 `sdata[0]` 与 `sdata[1]` 相加；当 `tid == 1` 时，则处理 `sdata[2]` 与 `sdata[3]`。这正是跨度为 `s = 1` 的第一轮归约过程。

2. 决定哪些线程参与计算的关键判断条件是：`if (tid % (2 * s) == 0)`，这条语句确保了每组参与计算的线程之间保持两倍跨度的间隔，从而避免冲突，并保证数据访问的连续性和高效性。

3. 在下一轮迭代中，跨度 `s` 增大为原来的两倍，即 `s *= 2`，此时每个线程将累加距离为 `s` 的两个元素。这一过程不断进行，直到所有线程的数据都被规约到一个值为止。最终当所有迭代完成后，只有 `tid == 0` 的线程将结果写入输出数组 `g_odata`，作为该线程块的局部归约结果。
   
   ```cpp
   __global__ void reduce_v0(float *g_idata, float *g_odata) {
     __shared__ float sdata[BLOCK_SIZE];
   
     unsigned int tid = threadIdx.x;
     unsigned int i = blockIdx.x * blockDim.x + threadIdx.x;
     sdata[tid] = g_idata[i];
     __syncthreads();
   
     for (unsigned int s = 1; s < blockDim.x; s *= 2) {
       if (tid % (2 * s) == 0) { // 相当于这里的tid = 2 * s
         sdata[tid] += sdata[tid + s];
       }
       __syncthreads();
     }
   
     if (tid == 0) g_odata[blockIdx.x] = sdata[0];
   }
   ```

从上述 CUDA 实现可以看出，该归约核函数采用的是分块归约（block-wise reduction）策略：每个线程块独立处理输入数据的一个子集，并将该子集的归约结果写入输出数组 `g_odata[blockIdx.x]`。例如，若输入数据总量为 1024*1024，我们配置了512个线程块，则每个线程块将负责 2048个元素的局部归约。

因此，在第一次调用 `reduce_v0` 后，我们会得到 512 个局部结果。为了最终获得整个数组的归约值，还需对这 512 个值进行第二次归约操作。通常的做法是再次调用相同的归约核函数，传入这 512 个中间结果作为新的输入，从而完成最终的全局归约。调用的代码如下，可以看出第一次我们分配了`num_blocks`个线程块进行计算，第二次再将`num_blocks`个线程块的局部和再进行二次归约得到最终的结果。

```cpp
reduce_v0<<<num_blocks, BLOCK_SIZE>>>(d_data, d_result);
reduce_v0<<<1, num_blocks>>>(d_result, d_final_result);
```

### 优化

#### 线程闲置

当前 kernel 存在一个显著的问题：在数据加载到共享内存后的归约阶段（即 `for` 循环部分），由于采用的是两两归约的方式，每次迭代中只有前一半线程参与计算，而后一半线程处于闲置状态。**随着迭代的进行，参与计算的线程数逐次减半，导致大量线程在整个归约过程中始终处于空闲状态，甚至有部分线程从始至终都没有执行过任何有效计算。**

#### 线程束分化

从性能分析的角度来看，`reduce_v0` 至少存在以下几个关键问题。首先，为了防止多个线程重复累加相同的数据，我们在循环中加入了条件判断：

```C++
if (tid % (2 * s) == 0)
```

这一机制虽然可以避免数据竞争，但也带来了严重的**线程束分化**问题。以第一次迭代为例，当跨度 `s = 1` 时，该条件等价于 `tid % 2 == 0`，即只有偶数索引的线程会参与计算，而奇数索引的线程则跳过执行。**由于一个 warp 通常包含连续的 32 个线程，这种条件判断将导致同一 warp 中的线程被划分为两个不同的执行路径：一部分线程执行加法操作，另一部分线程空转等待。**

#### 存储体冲突(bank conflict)

从代码结构可以看出，在每一轮归约（reduction）迭代中，多个线程会连续访问共享内存中的相邻元素。以 `s = 1` 的第一轮归约为例，当线程索引 `tid` 从 `0` 到 `num_threads - 1` 依次递增时，每个线程将访问 `sdata[tid]` 和 `sdata[tid + s]`。

```C++
if (tid % (2 * s) == 0) {   sdata[tid] += sdata[tid + s];}
```

在 CUDA 的共享内存架构中，共享内存被划分为多个称为 **bank**（存储体）的独立单元，通常为 32 个。每个 bank 在一个时钟周期内只能服务一个线程的访问请求。对于 32 位（4 字节）的数据类型（如 `float`），第 `i` 个线程访问的地址 `sdata[i]` 所属的 bank 编号为：

```C++
bank_id = (i * sizeof(float)) / 4 % 32 = i % 32
```

以当前访问模式为例，前提是不同线程对以下位置同时进行访问才构成bank冲突。

- `sdata[0]` 和 `sdata[32]` 位于同一个 bank；

- `sdata[1]` 和 `sdata[33]` 也位于同一个 bank；

当 `s = 16` 时，线程执行如下操作：

```C++
sdata[tid] += sdata[tid + 16];
```

当 `s == 16` 且 `tid == 0` 时，线程会访问 `s[0]` 和 `s[16]`；而当 `tid == 1` 时，则会访问 `s[32]` 和 `s[48]`。这种访问模式导致了 `s[0]` 和 `s[32]` 等地址之间出现 bank 冲突，因为它们映射到了相同的共享内存 bank。

#### 循环展开最后一个warp

1. 由于 **warp 是 GPU 的基本调度和执行单元**，在归约过程（即 `for` 循环）中，当参与计算的线程数减少到 32 或以下时，所有活跃线程将归属于同一个 warp。此时，这些线程在硬件层面是以 **SIMT（Single Instruction, Multiple Thread）** 模式同步执行的，因此无需再调用 `__syncthreads()` 进行同步。

2. 同时，在此阶段也不再需要使用条件判断语句如 `if (tid % (2 * s) == 0)` 或 `if (tid < s)` 来筛选参与计算的线程。因为在一个 warp 内部，所有线程都会被调度执行相同的指令，这种条件判断反而会导致 **warp 内部的控制流分化（warp divergence）**，影响执行效率。

3. 因此，在实现中我们可以在循环的后期阶段（即线程索引 `tid < 32` 时）对归约操作进行**手动展开**，直接执行对应的加法操作，从而消除分支判断和同步开销，提升性能。例如，可以将最后1轮的归约操作写为如下形式：

```C++
sdata[tid] += sdata[tid + 16];
sdata[tid] += sdata[tid + 8];
sdata[tid] += sdata[tid + 4];
sdata[tid] += sdata[tid + 2];
sdata[tid] += sdata[tid + 1];
```
